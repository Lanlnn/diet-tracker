# 开发复盘与防复发清单

本文记录 M0–M8 开发过程中已经实际发生的问题、根因和固定处理规则。它不是聊天记录，而是后续 M9、M10 和后台管理开发的开工检查表。

## 1. 2026-07-15 项目快照

- `master` 已包含 M0–M6，最新提交为 `323180a`，已有 `m0-complete`–`m6-complete` 标签。
- M7 已在本地分支 `codex/m7-exercise` 完成 3 个提交，但尚未推送、创建 PR、合并和打标签。
- M8 已在本地分支 `codex/m8-trends` 完成 3 个提交，它基于尚未合并的 M7，因此不能直接作为独立 M8 PR 合入 `master`。
- M8 自动化和微信开发者工具编译已通过；成功态页面仍需要连接可用本地后端完成整页截图和数据联调验收。
- 工作区还有与小程序里程碑无关的后台管理文档和原型改动，提交时必须继续按路径隔离。

## 2. 已发生的问题与固定规则

### 2.1 阶段依赖没有先合并

**现象：** M7 尚未进入 `master` 时开始 M8，导致 M8 分支同时包含 M7 和 M8 提交。直接向 `master` 创建 M8 PR 会把两个阶段混在一起，无法独立验收、回滚和打标签。

**固定规则：**

1. 阶段代码可以在明确的堆叠分支上提前开发，但交付必须严格按 M7 → M8 顺序。
2. 前置阶段 PR 合并后，后续阶段先变基到最新 `master`，确认 `git log master..HEAD` 只剩当前阶段提交，再创建 PR。
3. `mN-complete` 只能从合并后的 `master` 创建；本地完成或 Draft PR 都不能打完成标签。

### 2.2 工作区并行改动容易混入提交

**现象：** 开发 M8 时工作区同时出现 `admin-ui-ux-pro-max/`、`docs/ADMIN-DEVELOPMENT.md` 和 README 改动。使用 `git add -A` 会把无关功能带入里程碑提交。

**固定规则：**

- 开工和提交前都运行 `git status --short --branch`。
- 只用明确路径暂存，暂存后检查 `git diff --cached --check` 和 `git diff --cached --stat`。
- 不删除、不回滚来源不明的文件；无法隔离时停止提交并先确认归属。

### 2.3 本机 Java 与项目基线不一致

**现象：** 项目和 CI 使用 Java 17，但本机默认 Java 26。Mockito/Byte Buddy 在 Java 26 下报无法插桩，业务测试被误判为失败；切换到兼容 JDK 后测试通过。

**固定规则：**

```bash
java -version
cd backend
JAVA_HOME=$(/usr/libexec/java_home -v 17) sh mvnw --batch-mode test
```

- CI 的 Temurin 17 结果是交付基准。
- 遇到 Mockito/Byte Buddy “unsupported Java”时先核对运行时，不先修改业务代码或放宽测试。
- 新开发机如果没有 JDK 17，先安装并设置 `JAVA_HOME`，不要长期依赖更高版本碰巧兼容。

### 2.4 CI 只列举了早期测试

**现象：** 仓库已有 9 个小程序测试文件，旧 CI 只运行登录、资料和食品搜索 3 个；M4–M8 新增的计算、首页、餐次、运动和趋势测试不会在 PR 自动执行。

**固定规则：** CI 使用 `node --test miniapp/tests/*.test.js` 自动发现全部测试。新增测试不得再维护手写文件白名单。

### 2.5 Node 语法通过不代表开发者工具能编译

**现象：** 动态内联样式如 `style="width: {{value}}%"` 能通过 Node 语法检查，但微信开发者工具 Nightly 的生成 CSS 检查会报 `semi-colon expected`。M8 验收时发现并修复了首页遗留的两处问题。

**固定规则：**

- 提交前必须在微信开发者工具执行一次“普通编译”，问题面板必须为 0。
- 动态进度或柱高优先映射为有限 CSS class；确需动态样式时必须先用当前目标基础库和开发者工具验证。
- Node 测试、JSON/JS 静态检查、开发者工具编译是三道不同检查，互不替代。

### 2.6 自动化、联调和视觉验收曾被混为一个“完成”

**现象：** M8 的接口和流程测试全部通过，错误态也在开发者工具检查过，但因为本地后端未运行，成功态真实数据整页没有完成截图对照。此时只能标记“待验收”，不能标记“已完成”。

**固定规则：** 阶段完成必须分别记录：

1. 自动化：后端、前端、静态检查、真实 MySQL 迁移 CI。
2. 联调：小程序通过真实请求拿到符合契约的数据，并与相关页面对账。
3. 视觉：成功、空、错误等适用状态在目标模拟器或真机检查，并保存截图。
4. 交付：PR、CI、合并、标签。

四类证据没有齐全时，状态只能是`进行中`或`待验收`。

### 2.7 文档状态和架构描述会随代码漂移

**现象：** 里程碑曾出现“已完成（待验收合并）”“开发完成，待……”等自由文本，和规定的四种状态不一致；`architecture.md` 仍描述旧四 Tab、旧统计页和“Java 版本待统一”。

**固定规则：**

- 状态字段只写`未开始`、`进行中`、`待验收`、`已完成`；日期和说明另起一段。
- 每次修改导航、核心实体、接口或运行时基线时，同时更新 `architecture.md`、`api.md` 和当前里程碑文档。
- 开始下一阶段前运行：

```bash
rg -n '^\*\*状态：\*\*' docs/milestones
rg -n '待统一|旧版|V1–V[0-9]' docs README.md
```

### 2.8 产品决策已经实现，但决策清单没有回填

**现象：** 运动消耗可手动修正、用户目标采用手动设置等规则已经进入代码和 UI，`DEVELOPMENT.md` 仍标为未确认，后续开发者可能再次做出冲突实现。

**固定规则：** 产品规则一旦进入已合并代码，立即回填决策清单、PRD 或 ADR，并写明采用的口径；聊天结论不能作为唯一依据。

### 2.9 统计口径必须集中定义

**现象：** 旧统计页只统计饮食，新趋势页需要同时处理摄入、运动、净摄入、完整日期、目标达成率、前一周期和时区。若前后端各自推断，会出现首页与趋势对不上。

**固定规则：**

- 后端是统计口径唯一来源；前端只格式化和绘图。
- 饮食统计读取记录快照，运动统计读取保存后的最终消耗值。
- 日期边界使用配置时区和可注入 `Clock`，测试不得依赖运行机器的当天日期。
- 每个聚合接口至少有一条与明细或首页的对账测试。

### 2.10 本地数据库备份必须持续隔离

**现象：** 历史 MySQL 迁移会在 `backend/var/backups/` 生成真实数据库备份。当前目录已被 `.gitignore` 正确忽略，但其中可能包含用户数据，不能因为 Git 未跟踪就随意上传或附到 PR。

**固定规则：**

- `backend/var/` 永久保持忽略，不执行强制添加。
- 共享迁移证据只使用脱敏后的行数、校验和和日志摘要，不共享 SQL dump。
- 清理备份前先确认恢复窗口和数据保留要求，不自动删除。

## 3. M8 闭环退出条件

M8 当前是`待验收`。只有以下项目全部完成后才能改为`已完成`：

- [x] 后端统一趋势接口、完整日期、快照聚合、时区和规则总结完成。
- [x] 小程序 7/30/90 天切换、过期响应保护、无障碍摘要和状态处理完成。
- [x] 后端与小程序全量自动化通过。
- [x] 微信开发者工具普通编译问题面板为 0。
- [ ] M7 先创建 PR、CI 通过、完成视觉验收并合并，创建 `m7-complete` 标签。
- [ ] M8 变基到最新 `master`，确认 PR 只包含 M8 提交。
- [ ] 使用可用本地后端完成 M8 成功态、少于 3 天态和 30/90 天横向图表联调。
- [ ] 对照 `design/优化版/04-热量与运动趋势.png` 保存成功态截图并更新 `design-qa.md`。
- [ ] 推送 M8、创建 PR、CI 全绿、合并后创建并推送 `m8-complete` 标签。

## 4. 推荐的下一步顺序

1. **先收口 M7。** 不绕过依赖直接提交 M8；补齐 M7 成功态视觉证据，推送并创建 M7 Draft PR。
2. **让 M7 CI 全绿并合并。** 合并后从最新 `master` 创建 `m7-complete` 标签。
3. **整理 M8 分支。** 将 M8 变基到新 `master`，检查只剩 M8 的 3 个功能/测试提交和必要的质量修复。
4. **完成 M8 真实联调与截图。** 启动 Java 17 + MySQL 8 本地环境，准备至少 3 天饮食与运动测试数据，核对首页同日数据。
5. **提交 M8 PR 并闭环。** CI、视觉、文档都完成后合并和打标签。
6. **再开始 M9。** M9 首先定义整月摘要接口、月份时区边界和一次加载策略，不复用旧“历史”页逐日请求模式。

## 5. 每次开工的最小检查

```bash
git status --short --branch
git fetch origin
git log --oneline --decorate -8
java -version
(cd backend && sh mvnw --batch-mode test)
node --test miniapp/tests/*.test.js
find miniapp -name '*.js' -not -path '*/miniprogram_npm/*' -print0 | xargs -0 -n1 node --check
git diff --check
```

随后在微信开发者工具执行普通编译并检查问题面板。涉及页面交付时，必须同时准备可复现的成功态数据和设计稿对照截图。

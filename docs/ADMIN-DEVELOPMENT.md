# 轻养管理后台功能需求与开发执行文档

**版本：** v1.0

**状态：** A0 契约已冻结（变更须评审）

**日期：** 2026-07-15
**适用团队：** 产品、设计、管理后台前端、Java 后端、测试、运维

## 1. 文档目标

本文件定义与“轻养”微信小程序同步建设的管理后台。后台不是独立产品，也不照搬小程序页面；它为小程序已经存在或已经进入 M0–M10 计划的能力提供数据维护、问题排查、规则管理、运行监控和审计能力。

“前后台 1:1”在本项目中表示：

1. 每个后台功能必须能反查到小程序页面入口、PRD 业务规则、接口或上线保障需求。
2. 前台展示的系统数据和规则必须有明确的后台来源或代码来源，不建立第二套口径。
3. 后台不得直接修改用户历史快照、最终运动消耗等会破坏前台历史一致性的数据。
4. 前台尚未立项的业务，不得因为后台原型中出现就进入当前开发范围。
5. 小程序里程碑是后台功能开放的阶段门，不提前依赖尚未合并的模型和接口。

本文件的上游基准按优先级为：

1. `docs/饮食与运动小程序-产品需求文档-PRD-v1.0.md`
2. `design/优化版/页面索引.md` 与 8 张优化版页面稿
3. `docs/DEVELOPMENT.md` 和 `docs/milestones/M0-foundation.md`–`M10-profile-release.md`
4. `docs/api.md`
5. 当前后端代码、Flyway 迁移和小程序 Service

出现冲突时，先修正上游文档或完成产品决策，不允许后台自行创造兼容逻辑。

### 2.2 当前数据与接口

已存在并可复用的领域：

- 用户资料：`users`，`GET/PUT /api/users/me`。
- 系统食品与用户私有食品：`food_item`、`food_category`、`food_favorite`。
- 食品搜索、详情、计算、自定义、收藏：`/api/foods/**`。
- 饮食记录与营养快照：`meal_record`、`/api/records/**`。
- 今日聚合：`GET /api/dashboard/today?date=`。
- 运动记录与规则推荐：`exercise_record`、`/api/exercises/**`、`/api/exercise-recommendations`。
- 趋势：`GET /api/stats/trend?range=`。

尚未存在，不能被后台假设为已完成的能力：

- M9 月历摘要接口和月度模型。
- M10 独立 `UserGoal`、账号数据导出/删除流程、上传文件生命周期和健康检查。
- 产品埋点采集、搜索无结果聚合、首次记录漏斗和记录耗时指标。
- 管理员身份、角色权限、审计日志和管理专用 API。
- 可在线发布的运动系数、建议规则版本模型。

## 3. 范围与边界

### 3.1 当前必须建设

- 管理员独立登录、角色权限、会话安全和操作审计。
- 系统食品分类及系统食品数据维护。
- 营养计算预览，确保后台录入值与小程序计算结果一致。
- 用户、饮食、运动、趋势的只读问题排查视图；仅在对应里程碑可用后开放。
- 产品埋点和接口监控的数据基础；在数据就绪后建设运营总览。
- M10 上线前的基础健康状态、迁移版本和生产安全检查。

### 3.2 随小程序阶段同步建设

- M7 合并后：运动记录诊断、运动系数和推荐规则只读查看。
- M8 合并后：用户趋势诊断和首页/趋势对账。
- M9 合并后：月历诊断和月度请求性能监控。
- M10 合并后：目标诊断、隐私请求、账户删除审计、文件生命周期和健康检查。
- 埋点方案合并后：首次记录漏斗、记录耗时、搜索无结果率和留存分析。

### 3.3 当前明确不做

以下功能没有对应的小程序需求或数据闭环，不能进入当前后台版本：

- “申请公共收录”及公共食品投稿审核。
- 运营人员向用户发送提醒、通知或营销活动。
- 用户封禁、会员等级、付费、课程、社交和排行榜管理。
- 后台代用户新增、编辑或删除饮食和运动记录。
- 自由文本 AI 建议生成和 Prompt 管理。
- 不存在的“每日汇总任务”或定时任务控制台。
- 为拍照识别食物建设审核流程；V1 明确不做照片识别。

若未来需要上述功能，必须先补 PRD、前台入口、数据模型、隐私说明和独立里程碑，再更新本文件。

### 4.2 映射规则

- 系统食品由后台维护，用户自定义食品归用户所有，默认不进入公共库。
- 历史饮食统计永远读取记录快照；后台修改系统食品只影响后续计算，不回写历史。
- 运动消耗最终值由用户确认；后台不得批量重算历史运动记录。
- 首页、趋势和日历必须复用同一统计口径，后台诊断页只展示服务端结果和对账差异。
- 所有用户定位使用后台内部不可逆标识或脱敏展示，不把 `openid` 直接暴露给普通运营角色。

## 6. 详细功能需求

### 6.1 ADM-AUTH 管理员身份与权限

管理端不得复用小程序 JWT。管理员使用独立身份表、独立鉴权入口、独立 Token audience 和 `/api/admin/**` 路由。

角色建议：

| 角色 | 能力 |
| --- | --- |
| `SUPER_ADMIN` | 管理员、角色、系统配置和全部审计 |
| `FOOD_EDITOR` | 系统食品和分类维护、计算预览 |
| `SUPPORT_VIEWER` | 用户问题只读诊断，不查看敏感原文 |
| `RULE_PUBLISHER` | 规则草稿、校验、发布和回滚 |
| `ANALYST` | 聚合指标和脱敏明细只读 |

验收：

- 未授权访问返回 401，权限不足返回 403。
- 登录失败限速，管理员会话支持主动失效。
- 写操作记录操作者、时间、对象、变更前后摘要、原因和请求 ID。
- 权限不只控制菜单，还必须由后端接口二次校验。

### 6.2 ADM-FOOD 系统食品与分类

#### 分类列表

字段：名称、图标标识、排序、系统食品数量、更新时间。支持新增、编辑、排序；删除前检查食品引用，无安全迁移方案时禁止删除。

#### 系统食品列表

仅管理 `user_id IS NULL` 且来源属于系统的数据。字段：

- 名称、分类、来源。
- `baseAmount`、`baseUnit`。
- 热量、蛋白质、脂肪、碳水。
- `servingAmount`、`servingUnit`。
- 更新时间和最后操作人。

筛选：关键词、分类、营养基准单位、来源、更新时间。默认分页，不一次加载全量。

#### 新增与编辑

- 重量类默认每 100g；按份食品必须明确 `baseAmount=1` 和份单位。
- 数值使用十进制，禁止负值；名称必填且长度不超过当前接口约束。
- 保存前展示“输入 150g 的计算预览”，计算必须调用与小程序相同的后端服务。
- 编辑前提示“只影响未来计算，历史记录使用快照不变”。
- 首期不提供硬删除；如需下架，先在领域模型中增加可见性状态并评审小程序搜索行为。

#### 用户自定义食品诊断

支持按脱敏用户标识和食品 ID 查询，仅用于回答“为什么搜索不到/为什么计算不同”。默认隐藏备注和个人信息，不允许转为公共食品，不允许代用户编辑。

### 6.3 ADM-SUPPORT 用户问题诊断

这是客服和开发排障工具，不是用户运营 CRM。

#### 用户检索

当前 A3-1 仅接受稳定、随机、唯一的 `supportRef` 精确查询，不提供昵称、openid 或其他个人资料模糊搜索。结果展示脱敏编号、创建时间和基础目标资料，不提供封禁状态。

#### 用户详情

按前台结构组织标签：

1. 今日：指定日期的摄入、目标、剩余、运动、营养和餐次摘要。
2. 餐次：记录输入量、单位、营养快照、客户端幂等键和时间。
3. 运动：时长、强度、估算来源和用户最终确认消耗。
4. 趋势：7/30/90 天服务端返回值和数据不足状态。
5. 日历：M9 后显示月摘要和请求耗时。
6. 目标与隐私：M10 后显示目标版本、删除请求状态和处理结果。

所有页面为只读。后台展示的统计值必须直接来自对应领域服务或同一聚合查询，不在管理端重新计算。

#### 对账工具

- 今日摄入等于当天有效饮食记录快照之和。
- 今日运动等于当天运动记录最终消耗之和。
- 净摄入等于摄入减运动。
- 剩余可摄入为 `max(目标-摄入, 0)`，不返还运动热量。
- 差异出现时展示差异项、请求 ID 和时间范围，不提供“一键修正历史”。

### 6.4 ADM-RULE 规则配置

当前运动系数和推荐规则仍由代码/配置实现。首期后台只读展示生效规则及来源，不直接在线修改。

需要在线发布前，必须先增加：

- `rule_set`：类型、版本、状态、适用范围、生效时间。
- `rule_item`：条件、结果、优先级和安全文案。
- `rule_release`：发布人、审批人、发布时间和回滚版本。

发布流程固定为：草稿 → 自动校验 → 预览样例 → 审批 → 定时/立即生效 → 可回滚。不得直接编辑已发布版本。

规则边界：

- 运动推荐最多两项。
- 不输出医学诊断、治疗或结果承诺。
- 用户可修改估算消耗，规则发布不重算历史。
- 场景建议只基于已记录数据；数据不足时不显示。
- 本模块不包含向用户主动发送消息的能力。

### 6.5 ADM-ANALYTICS 产品质量

本模块只有在埋点和接口监控落地后开放，不能从业务表猜测用户行为。

核心指标与来源：

| 指标 | 定义 | 数据来源 |
| --- | --- | --- |
| 首次记录完成率 | 首次进入记录链路后成功保存的用户数/进入用户数 | `record_entry_click`、`meal_record_success` |
| 单次记录中位耗时 | 进入记录链路到成功保存的 P50 | 同一匿名会话事件时间 |
| 7 日记录留存 | 首次成功记录后第 7 日仍成功记录 | `meal_record_success` |
| 搜索无结果率 | 无结果搜索次数/有效搜索次数 | `food_search`、`food_search_empty` |
| 首页接口成功率 | 成功请求/全部首页聚合请求 | 服务端监控 |

事件至少包含：事件时间、页面、匿名用户标识、来源入口、结果状态、请求 ID 和事件版本。不得采集搜索全文、备注全文、昵称、体重或图片地址作为通用分析维度。

图表规则：

- 时间趋势使用折线图，并提供数据表。
- 首次记录链路使用 3–8 步漏斗，显示每步人数和转化率。
- 指标对目标使用紧凑型进度/子弹图，数值与目标始终可见。
- 图表不能只靠颜色表达；异常点同时使用形状、文字和列表。

### 6.6 ADM-DASH 工作台

在产品质量数据就绪前，工作台只展示可验证的运行状态：当前小程序里程碑、后端版本、数据库迁移版本、关键接口健康和最近高风险审计操作。

埋点就绪后再增加五个 PRD 指标。日活、总用户数和总记录数可以作为辅助信息，但不得替代 PRD 成功指标。

### 6.7 ADM-PRIVACY 隐私请求（M10）

- 账号删除只能由用户在小程序发起并完成必要确认。
- 后台展示请求状态、处理范围、失败原因和审计证据，不提供普通运营任意删除用户的按钮。
- 完成删除后，审计记录使用不可逆用户摘要，不保留可恢复个人数据。
- 数据导出必须限权、限时、加密并记录下载审计。
- 图片保留数量、大小、保存周期和删除策略未决前，不开发照片管理页面。

### 6.8 ADM-SYSTEM 系统与安全

- 管理员和角色管理。
- 审计日志按操作者、对象、动作、日期和请求 ID 检索。
- 展示应用版本、环境、数据库连接状态和 Flyway 版本。
- 监控首页、搜索、保存、运动、趋势、月历接口的成功率与延迟。
- 生产环境检测并禁止 Seed、调试接口和弱密钥配置。
- 不存在的定时任务不显示在后台；新增真实任务后再纳入监控。

## 7. 管理 API 与数据原则

### 7.1 路由

所有后台接口使用 `/api/admin/**`。建议首批接口：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| POST | `/api/admin/auth/login` | 管理员登录 |
| POST | `/api/admin/auth/logout` | 会话失效 |
| GET | `/api/admin/foods` | 系统食品分页 |
| POST | `/api/admin/foods` | 新增系统食品 |
| PUT | `/api/admin/foods/{id}` | 编辑系统食品 |
| POST | `/api/admin/foods/preview` | 同口径营养计算预览 |
| GET | `/api/admin/food-categories` | 分类列表 |
| POST/PUT | `/api/admin/food-categories/**` | 分类维护 |
| GET | `/api/admin/support/users` | 脱敏用户检索 |
| GET | `/api/admin/support/users/{id}/today` | 单日诊断 |
| GET | `/api/admin/support/users/{id}/records` | 饮食快照诊断 |
| GET | `/api/admin/support/users/{id}/exercises` | 运动诊断，M7 后 |
| GET | `/api/admin/support/users/{id}/trend` | 趋势诊断，M8 后 |
| GET | `/api/admin/support/users/{id}/calendar` | 月历诊断，M9 后 |
| GET | `/api/admin/audit-logs` | 审计查询 |
| GET | `/api/admin/system/health` | 健康与版本，M10 前上线准备 |

### 7.2 复用规则

- 后台 Controller 可以有独立 DTO，但必须调用现有领域 Service，不复制营养和统计公式。
- 管理员上下文与用户上下文分离；后台查询用户数据必须显式校验权限并写审计。
- 分页、排序和筛选在服务端执行。
- 所有日期按项目配置时区解析，接口返回明确时区。
- 错误响应沿用 `code`、`message`、`requestId` 和字段错误结构。
- 数据结构变化只通过 Flyway；不在管理端自动建表或改表。

## 8. 状态、交互与视觉要求

后台视觉继承小程序品牌：暖米白背景、深苔绿主色、低饱和橙黄辅助色。通用后台模板给出的蓝色方案不采用。桌面端以高信息密度和可扫描性为主，不复刻移动端 24px 大圆角和大面积卡片。

### 8.1 页面状态

每个异步页面必须区分：

```text
idle / loading / success / empty / error / submitting / forbidden
```

- 加载超过 300ms 显示骨架或进度。
- 空状态说明为什么为空以及可执行的下一步。
- 错误状态包含原因、请求 ID 和重试入口。
- 写按钮提交期间禁用，成功或失败均给出明确反馈。
- 只读字段和禁用字段在视觉及语义上必须可区分。

### 8.2 桌面交互

- 1024px 以上使用侧边栏；窄屏折叠导航，表格允许受控横向滚动或切换卡片摘要。
- 所有核心页面有稳定 URL，支持深链接和刷新恢复筛选条件。
- 表格表头可排序时使用明确状态和 `aria-sort`。
- 键盘可操作，焦点环可见；图标按钮必须有文本标签或 `aria-label`。
- 正文对比度至少 4.5:1，状态不只依靠红绿颜色。
- 图标统一使用同一套 SVG，不使用 Emoji 充当结构图标。
- 动效仅用于状态反馈，150–300ms，并尊重 `prefers-reduced-motion`。

### 8.3 高风险操作

- 规则发布、角色调整等高风险操作必须二次确认并填写原因。
- 确认框明确展示对象、影响范围和是否可回滚。
- 系统食品编辑显示历史快照不受影响的提示。
- 首期没有后台删除用户历史记录的操作。

## 9. 分阶段开发计划

### A0 冻结产物

- 管理 API 契约：[`admin/openapi.yaml`](./admin/openapi.yaml)
- 管理员身份、RBAC、会话及审计决策：[`adr/ADR-001-admin-security-audit.md`](./adr/ADR-001-admin-security-audit.md)
- PRD 指标事件口径：[`admin/analytics-events.md`](./admin/analytics-events.md)

冻结规则：第 4 节映射、上述 OpenAPI 路由、角色权限或审计字段发生不兼容变更时，必须同步更新三份产物并记录评审结论；后台页面不得自行扩展当前范围。

### A0：契约冻结与安全设计

依赖：无，可与小程序收口并行。

- 冻结本文件的前后台映射表。
- 决定管理员身份来源、角色和部署边界。
- 输出管理 API OpenAPI 草案和审计 ADR。
- 为 PRD 五项指标定义事件口径，但不先做假数据看板。

完成标准：每个需求都有前台 ID、数据来源、权限和阶段门。

### A1：管理员安全基础与前端骨架

依赖：A0。

- 管理员登录、会话、RBAC、审计拦截器。
- 侧边导航、路由、权限守卫和统一状态组件。
- 品牌 Token、表格、筛选、表单、确认框和错误页。

### A2：系统食品后台 MVP

依赖：A1；复用 M3–M5 已有领域能力。

- 分类维护、系统食品分页、新增和编辑。
- 同口径营养计算预览。
- 用户自定义食品只读诊断。
- 历史快照不回写的回归测试。

这是第一个可以独立交付业务价值的后台版本。

### A3：只读用户问题诊断

依赖：A1、M6；运动/趋势标签分别等待 M7/M8 合并。

- 脱敏用户检索。
- 今日、餐次和快照诊断。
- M7 后加入运动；M8 后加入趋势和对账。
- 不实现用户封禁和代用户修改数据。

### A4：埋点与产品质量数据基础

依赖：PRD 事件口径评审。

- 小程序事件 SDK/封装、服务端接收或既定分析平台接入。
- 首页、搜索、保存关键接口成功率和延迟监控。
- 数据质量校验、事件版本和隐私检查。

### A5：产品质量工作台

依赖：A4 产生稳定数据。

- PRD 五项指标、趋势和首次记录漏斗。
- 搜索无结果词只做合规聚合，禁止暴露用户敏感全文。
- 指标口径说明、更新时间和数据异常状态。

### A6：M9/M10 同步能力

依赖：对应小程序阶段合并。

- M9 月历诊断和性能监控。
- M10 目标诊断、隐私请求、文件生命周期和健康检查。

### A7：规则配置化

依赖：M7 验收、规则模型和安全文案评审。

- 先只读展示当前规则。
- 再实现版本、审批、发布、灰度和回滚。
- 上线前用固定样例验证推荐数量、安全边界和历史不重算。

### A8：上线准备

- 权限矩阵、审计完整性和越权测试。
- 管理 API 限流、日志脱敏、监控和告警。
- 数据库迁移与回滚演练。
- 1024/1280/1440 宽度、键盘和基础无障碍验收。
- 确认生产环境无 Seed/调试接口。

## 10. 测试与完成定义

### 10.1 核心契约测试

- 后台 150g 鸡胸肉预览与小程序结果一致。
- 修改系统食品后，旧饮食记录快照和趋势不变化。
- 后台今日诊断与小程序同用户同日期首页一致。
- 运动消耗不增加剩余可摄入。
- 趋势、日历与首页对相同日期对账一致。
- 普通支持角色无法调用食品写入、规则发布和管理员接口。
- 用户数据查询产生审计日志，且普通页面不暴露 openid。

### 10.2 完成定义

一个后台功能只有同时满足以下条件才算完成：

- 在第 4 节映射表中有明确前台来源和阶段门。
- 对应小程序里程碑已合并，不依赖本地未交付分支。
- 管理 API、权限、审计、校验和错误结构已完成。
- loading、success、empty、error、forbidden 和适用的 submitting 状态齐全。
- 关键读写、越权和跨端对账有自动化测试。
- 文档、OpenAPI、迁移和代码在同一功能 PR 更新。
- 桌面视觉、键盘操作和基础可访问性验收通过。

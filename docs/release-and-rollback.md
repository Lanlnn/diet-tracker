# 本地回退与未来发布手册

## 1. 当前适用范围

项目完成前只执行本地开发、局域网真机联调和本地数据保护。不部署 staging/production，不配置域名或 HTTPS，不上传正式体验版灰度。当前回退对象只有代码、Docker 容器、本地 MySQL 卷和头像卷。

## 2. 代码回退

- 每次阶段开始前记录 Git SHA；不在脏工作树上执行破坏性命令。
- 需要比较旧版本时新建临时 worktree，不覆盖用户未提交改动。
- Flyway 迁移和应用代码必须配套回退；禁止只切旧代码继续写入更高版本数据库。
- M0–M10 稳定基线为 `m10-complete`。

## 3. 本地服务操作

```bash
# 停止，保留数据
bash deploy/local/stop.sh

# 重新构建并启动
bash deploy/local/start.sh

# 仅在明确不需要本地数据时重置
bash deploy/local/reset-data.sh --confirm
```

`reset-data.sh` 只删除 `diet-tracker-local` 的 MySQL 和头像 Docker 卷，但操作不可逆。含真实测试数据时，先用 `mysqldump` 备份到 Git 忽略目录并记录校验和。

## 4. 数据迁移失败

1. 停止后端，保留失败日志、Git SHA 和 Flyway 错误，不记录密码或用户隐私。
2. 不手工修改 `flyway_schema_history`，不反复重试破坏性 SQL。
3. 对无价值的纯测试库可确认后重置卷，再由 Flyway 从空库重建。
4. 对需保留的数据使用迁移前备份恢复，随后在副本验证升级。
5. V8 的条件回滚脚本只用于已核对无新目标/删除审计数据的受控场景。

## 5. 本地回退触发条件

- 健康检查失败或后端持续重启。
- Flyway 失败、基础食品数量异常或跨页统计口径不一致。
- 登录出现用户串号、越权或 Token 泄露风险。
- 饮食/运动增删改导致重复写入或数据丢失。

发生上述情况时停止继续录入，保存脱敏 requestId 和复现步骤，修复后重跑完整主链路。

## 6. 项目完成后的云端发布门禁

只有用户明确启动云端阶段后才执行：

1. 选择云厂商和区域，设计 staging/production 隔离拓扑。
2. 使用 Secret 管理注入数据库、微信、JWT 和审计配置。
3. 配置公网域名、HTTPS 证书链、微信 request/uploadFile 合法域名。
4. 备份 MySQL 8，记录镜像 digest、Git SHA、Flyway 版本和恢复演练。
5. 上传微信体验版，在 iOS/Android 完成公网回归。
6. 按 5%→25%→100% 灰度，观察错误率、P95、401/429 和连接池。
7. 准备上一稳定镜像、数据库恢复点和停止灰度开关。

未来云端部署文件应在该阶段基于最终架构重新创建，不能直接假定当前局域网配置可用于生产。

# ADR-001：管理端身份、RBAC、会话与审计

- 状态：已接受
- 日期：2026-07-15
- 适用阶段：A0–A2

## 决策

1. 管理员存放于独立的 `admin_user` 表，不复用小程序 `users`、`openid` 或小程序 JWT。
2. 管理 Token 使用独立密钥、`aud=diet-tracker-admin`、短时有效期和唯一 `jti`。请求同时校验管理员启用状态与 `session_version`；退出登录或管理员会话失效时递增版本，使旧 Token 立即失效。
3. 首期角色固定为 `SUPER_ADMIN`、`FOOD_EDITOR`、`SUPPORT_VIEWER`、`RULE_PUBLISHER`、`ANALYST`。A2 食品写接口仅允许 `SUPER_ADMIN`、`FOOD_EDITOR`；食品读和预览亦只向这两类角色开放。
4. `/api/admin/**` 由独立过滤器处理，未登录返回 401，已登录但越权返回 403。后端权限是最终边界，前端菜单守卫仅改善体验。
5. 登录以用户名和密码完成；密码仅保存 BCrypt 哈希。按用户名与来源地址进行内存窗口限速，连续失败达到阈值后暂时拒绝，成功登录清空失败计数。
6. 管理写操作由审计拦截器统一记录。日志包含操作者、角色、动作、对象类型/ID、请求 ID、原因、结果及变更前后摘要；密码、Token 和敏感原文不得进入审计。
7. 管理端与 API 默认同域部署；开发环境可由显式 CORS 白名单访问。生产环境不得使用通配来源、默认管理员密码或弱管理密钥。

## 数据边界

- 管理端只编辑系统食品（`food_item.user_id IS NULL`），不得写用户私有食品。
- 系统食品编辑不更新 `meal_record` 营养快照。
- 分类存在食品引用时禁止删除；A2 不提供系统食品硬删除。
- 用户私有食品诊断只返回脱敏用户摘要和营养字段，不返回 `openid`、备注或个人资料。

## 初始管理员

迁移只创建表，不写默认账号。首次管理员通过显式环境变量和一次性启动引导创建；生产环境缺少安全配置时不自动 Seed。

## 后果

- 管理端可以独立失效会话，不影响小程序用户。
- 角色为单角色字段，若未来需要多角色组合，应通过新迁移和兼容契约演进。
- 当前限速器适合单实例；扩容为多实例前须迁移到共享存储或网关限速。

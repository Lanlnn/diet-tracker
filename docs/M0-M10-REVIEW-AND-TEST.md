# M0–M10 成果复盘与整体验收手册

本文用于集中展示 M0–M10 的交付效果，并指导一次可留存证据的整体验收。它不替代各阶段里程碑；阶段范围以 [`milestones/`](./milestones/README.md) 为准，接口以 [`api.md`](./api.md) 和实际代码为准。

## 1. 本轮复盘目标

1. 用一条完整用户旅程呈现 M0–M10，而不是逐页孤立演示。
2. 分开记录自动化、真实接口联调、开发者工具视觉和真机兼容性证据。
3. 对账饮食、首页、运动、趋势、日历和目标，确认它们使用同一套数据口径。
4. 暴露发布前缺口；未验证项目不得因为“页面可见”而勾选完成。

## 4. 复盘演示前准备

### 4.1 环境

- 使用 Java 17，并先确认命令输出确实包含 `version "17`，不能依赖 macOS 在找不到目标版本时的静默回退。
- 使用 MySQL 8 测试库，记录 Flyway 升级前后版本和行数。
- 后端使用 `local` 或专用验收环境；小程序 Network 必须指向该环境。
- 准备一个可保留数据的主验收账号，以及一个只用于账号删除的临时账号。
- 微信开发者工具执行普通编译，问题面板清零后再开始截图。

推荐先执行：

```bash
cd /Users/z/Documents/微信小程序/diet-tracker
JAVA_HOME=$(/usr/libexec/java_home -v 17 2>/dev/null)
"$JAVA_HOME/bin/java" -version 2>&1 | rg 'version "17[.]' || {
  echo "未找到可用的 Java 17"
  exit 1
}
(cd backend && JAVA_HOME="$JAVA_HOME" sh mvnw --batch-mode test)
node --test miniapp/tests/*.test.js
find miniapp -name '*.js' -not -path '*/miniprogram_npm/*' -print0 | xargs -0 -n1 node --check
bash backend/scripts/check-release-readiness.sh
git diff --check
```

### 4.2 统一演示数据

为了让页面之间可以对账，建议固定使用以下数据：

- 每日热量目标：1,800 千卡。
- 午餐：鸡胸肉 150g；基准为 165 千卡/100g，结果应为 248 千卡。
- 运动：快走 20 分钟；记录页面展示服务端保存后的最终消耗。
- 日期：今天至少有一条饮食和一条运动；前 6 天补充少量记录，用于展示 7 天趋势。

不要在共享数据库中使用真实健康信息；截图前检查昵称、头像、Token、openid 和请求正文。

## 5. 一条完整的成果演示路线

按下面顺序演示，可以把 11 个里程碑串成一条闭环。

### 场景 A：工程、登录与应用骨架（M0–M2）

1. 冷启动小程序，确认只发生一次登录流程。
2. 依次切换“今日、趋势、记录、运动、我的”五项导航。
3. 展示加载、无数据和断网错误状态，以及恢复网络后的重试。
4. 在“我的”修改昵称或头像，接口失败时不得先显示保存成功。

证据：开发者工具 Console/Network 脱敏截图、五 Tab 整页截图、一次失败重试录像或截图。

### 场景 B：食品发现与热量计算（M3–M4）

1. 从“今日”的“记录一餐”进入食品搜索。
2. 快速连续输入两个关键词，确认旧请求不会覆盖新结果。
3. 搜索“鸡胸肉”，进入计算页并输入 150g。
4. 核对热量为 248 千卡、蛋白质按同一比例计算。
5. 搜索不存在的食品，确认可以进入创建自定义食品流程。

证据：搜索正常/空/错状态、150g 计算结果、Network 中服务端复核响应。

### 场景 C：饮食记录与今日首页（M5–M6）

1. 连续点击保存，确认只产生一条午餐记录。
2. 返回首页，确认午餐详情和今日摄入同时增加 248 千卡。
3. 编辑份量后，核对餐次详情和首页同步变化。
4. 删除记录并二次确认，核对两个页面同步减少。
5. 为后续场景重新保存 150g 鸡胸肉记录。

证据：保存前后记录 ID、首页/餐次并排截图、重复提交时的请求数量。

### 场景 D：运动与跨模块热量口径（M7–M8）

1. 添加 20 分钟快走，展示最终运动消耗和最多两条建议。
2. 返回首页，确认运动消耗更新，但“剩余可摄入”没有因运动增加。
3. 打开趋势，依次切换 7/30/90 天。
4. 对今天核算：净摄入必须等于饮食摄入减运动消耗。
5. 检查少于 3 个有效记录日时的提示状态。

证据：首页、运动详情、趋势同日数值对账表和 7/30/90 天截图。

### 场景 E：月历与目标贯通（M9–M10）

1. 打开饮食日历，确认有记录日期的标记正确。
2. 切换上月、下月，确认每个月只请求一次月摘要，而不是逐日请求。
3. 选择今天，核对摘要与首页同日期数据一致。
4. 在“我的”把热量目标改为另一个合法值，重新打开首页和趋势，确认目标口径同步。
5. 展示个人中心的自定义食品、收藏、本周运动次数和连续记录天数摘要。
6. 展示规则化 AI 教练开关，不输出自由文本医学建议。

证据：月历 Network 请求、选中日/首页对账、目标修改前后页面截图。

### 场景 F：隐私、安全与发布准备（M10）

1. 打开设置、隐私和关于页面，确认删除账号有明确的不可逆提示和二次确认。
2. 只使用临时账号执行删除；删除后原 Token 不得继续访问用户接口。
3. 核对饮食、运动、目标、自定义食品、收藏和托管头像已按设计清理。
4. 核对审计只保留匿名散列、事件 ID、requestId、结果和时间。
5. 验证生产 Profile 下 Seed 不可访问、健康检查可用、限流开启。
6. 按 [`release-and-rollback.md`](./release-and-rollback.md) 演练 V8 升级和回滚。

账号删除不可逆，必须放在整场复盘最后，并使用专用临时账号。

## 6. 异常与边界测试矩阵

| 类别 | 必测项 | 预期 |
| --- | --- | --- |
| 网络 | 首屏断网、保存超时、恢复后重试 | 已成功数据不被清空；不假成功；可重试 |
| 鉴权 | Token 过期、多个请求同时 401 | 只刷新一次；失败后回到可恢复状态 |
| 输入 | 空值、负数、超范围、多位小数 | 前后端规则一致，返回结构化字段错误 |
| 幂等 | 饮食/运动快速重复提交 | 只生成一条业务记录 |
| 隔离 | 访问他人自定义食品、饮食或运动 ID | 返回不可见或拒绝，不泄露存在性和内容 |
| 日期 | 大小月、闰年、跨年、时区日界线 | 月历、首页、趋势使用同一天界线 |
| 聚合 | 编辑/删除后查看首页、趋势、月历 | 所有聚合刷新后一致 |
| 隐私 | 删除失败、头像删除失败、审计查看 | 业务事务不出现半删除；日志不含敏感字段 |
| 发布 | 缺失生产变量、Seed、限流、健康检查 | 配置缺失明确失败；生产安全开关符合要求 |

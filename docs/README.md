# 项目文档索引

本文档目录用于区分产品目标、设计基准、开发流程和当前实现，避免旧版原型与优化版需求混用。

## 阅读顺序

1. [从这里开始开发](./START-HERE.md)：确认当前里程碑、本地环境和统一交付循环。
2. [M0–M10 成果复盘与整体验收手册](./M0-M10-REVIEW-AND-TEST.md)：按完整用户旅程演示成果、执行测试并留存证据。
3. [产品需求文档](./饮食与运动小程序-产品需求文档-PRD-v1.0.md)：确认产品范围、业务规则和验收标准。
4. [优化版页面索引](../design/优化版/页面索引.md)：确认页面关系和跳转路径。
5. [优化版设计说明](../design/优化版/设计说明.md)：确认视觉和信息层级。
6. [开发执行指南](./DEVELOPMENT.md)：按阶段实施前端、后端、数据和测试工作。
7. [M0–M10 里程碑](./milestones/README.md)：每阶段的任务、验收和 Git 交付。
8. [系统架构](./architecture.md)：了解当前系统结构；后续架构调整以开发指南为准。
9. [API 文档](./api.md)：记录当前已实现接口；接口变化必须与代码同步更新。

## 唯一基准

- 产品范围：`docs/饮食与运动小程序-产品需求文档-PRD-v1.0.md`
- UI 与交互：`design/优化版/`
- 开发顺序和质量门槛：`docs/DEVELOPMENT.md`
- 实际接口行为：以后端代码和自动生成的 OpenAPI 为准

旧版 `design/` 根目录下的 20 张页面稿仅用于追溯功能来源，不作为新页面实现依据。

## 文档维护规则

- 产品规则变化：先更新 PRD，再修改代码。
- 页面结构变化：同时更新优化版页面索引和开发指南。
- API 字段变化：代码、API 文档和前端类型说明必须在同一次提交中更新。
- 数据库变化：必须通过迁移脚本记录，不直接手工修改生产表。
- 已完成事项使用 Git 提交和任务记录跟踪，不在文档中写无法验证的“已完成”。

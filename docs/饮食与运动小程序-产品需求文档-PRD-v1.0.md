# 饮食与运动小程序产品需求文档（PRD）

**版本：** v1.0
**状态：** 开发评审稿
**日期：** 2026-07-14
**适用团队：** 产品、UI、微信小程序前端、Java 后端、测试

## 1. 文档目的

本文档用于统一饮食与运动小程序优化版的产品范围、计算口径、页面行为、数据结构、接口需求及验收标准，作为开发排期、技术评审和测试用例设计的依据。

对应设计稿位于：`design/优化版/`。

## 2. 产品背景与目标

现有版本已具备食物库、饮食记录、历史记录和基础统计能力，但页面入口较多，饮食、统计、训练、AI 教练和个人服务相互并列，主任务不突出。优化版将产品核心收敛为“看懂今天的热量状态，并完成一次准确记录”。

### 2.1 产品目标

1. 用户进入首页后 3 秒内知道今天已摄入、剩余可摄入和运动消耗。
2. 用户可在 30 秒内完成一次常用食物的热量计算与记录。
3. 用户可以记录当天运动，并获得最多两个简单、可执行的运动建议。
4. 饮食与运动在趋势页中形成同一套热量收支视图。
5. 将原 20 个设计页面收敛为 8 个核心页面，低频功能下沉至个人中心。

### 2.2 成功指标（上线后观察）

- 首次记录完成率 ≥ 70%。
- 单次饮食记录中位耗时 ≤ 30 秒。
- 7 日记录留存率较现版本提升 15%。
- 食物搜索无结果率 ≤ 10%。
- 首页关键数据接口成功率 ≥ 99.5%。

### 2.3 本期不做

- 不通过照片自动识别食物或估算热量；拍照仅做留存。
- 不接入可穿戴设备或微信运动自动同步。
- 不提供医疗诊断、疾病治疗或极端减重建议。
- 不建设复杂社交、排行榜、课程商城或付费体系。
- AI 教练 V1 只输出规则化建议，不直接调用大模型生成自由文本。

## 3. 目标用户与核心场景

### 3.1 目标用户

- 希望减脂、维持体重或改善饮食结构的普通用户。
- 已有基础饮食意识，但不熟悉复杂营养学计算。
- 需要快速记录，不愿填写大量表单。

### 3.2 核心场景

1. 早餐后搜索“鸡蛋”，选择重量并加入早餐。
2. 午餐前查看今天还可以摄入多少热量。
3. 晚饭后记录快走 20 分钟及消耗热量。
4. 查看本周饮食摄入、运动消耗和净摄入趋势。
5. 修改每日热量目标或维护个人常用食品。

## 5. 核心业务规则

### 5.1 食物营养数据标准

V1 食品库统一使用“每 100 克”作为重量类食物的营养基准。必须新增或确认以下字段：

- `baseAmount`：营养基准数量，默认 100。
- `baseUnit`：营养基准单位，默认 `g`。
- `calories`、`protein`、`fat`、`carbs`：对应基准数量的营养数据。
- `servingAmount`、`servingUnit`：可选的一份重量及展示单位。

已有数据如果按“每份”存储，必须在迁移时标记或转换，不能同时按每份和每 100g 混算。

### 5.2 单项食物计算

重量类食物：

`本次热量 = 每100g热量 × 食用克数 ÷ 100`

`本次营养素 = 每100g营养素 × 食用克数 ÷ 100`

计算过程保留至少 4 位小数，展示时热量四舍五入为整数，营养素保留 1 位小数。后端保存原始输入重量及计算快照，避免食品库后续修改导致历史记录变化。

### 5.3 每日热量口径

- `饮食摄入 = 当日所有有效饮食记录热量之和`
- `运动消耗 = 当日所有有效运动记录消耗热量之和`
- `剩余可摄入 = max(每日饮食目标 - 饮食摄入, 0)`
- `净摄入 = 饮食摄入 - 运动消耗`

V1 默认不把运动消耗自动增加到“剩余可摄入”额度中。首页分别展示剩余可摄入与运动消耗；趋势页展示净摄入。后续如需开放“运动热量返还”，必须做成用户可配置项并重新评审。

### 5.4 目标与安全边界

- 每日热量目标由用户设置或按运营规则推荐，V1 不自动进行医学化计算。
- 目标值建议范围为 1,000–5,000 千卡，超出范围需二次确认。
- 建议文案不得承诺具体减重结果，不得使用诊断性语言。

### 5.5 运动消耗

- V1 支持用户手动选择运动类型、时长和强度。
- 系统根据运动类型、时长与配置系数给出预估消耗，用户可修改。
- 最终以用户确认并保存的 `caloriesBurned` 为准。
- 运动推荐最多展示两个，优先选择低门槛、短时长、符合用户目标的内容。

## 6. 页面功能需求

### 6.1 今日首页（P0）

**页面目标：** 让用户快速理解今日热量状态，并进入记录流程。

**功能要求：**

- 展示日期、饮食摄入、每日目标、剩余可摄入、运动消耗。
- 展示碳水、蛋白质、脂肪的摄入量及目标完成度。
- 展示今日已完成运动数量、时长和消耗。
- 展示最多一条可执行建议。
- 展示早餐、午餐、晚餐和加餐摘要；未记录餐次显示明确入口。
- “记录一餐”进入食品搜索；相机按钮进入拍照留存。
- 日历按钮进入饮食日历。

**异常状态：**

- 无记录：展示空状态和“记录第一餐”。
- 部分接口失败：已成功模块正常展示，失败模块提供重试。
- 超出目标：剩余显示 0，并展示“今日超出 X 千卡”，不得显示负数。

### 6.2 食品搜索与记录（P0）

- 支持按食物名称模糊搜索，输入 300ms 后触发查询。
- 默认展示常用、最近、收藏三个标签；V1 可先实现常用与最近。
- 支持拍照留存、自定义食物和最近组合入口。
- 搜索结果展示名称、基准热量、基准单位和添加按钮。
- 无结果时提供“创建自定义食物”。
- 选择食物后进入热量计算页。

### 6.3 热量计算（P0）

- 展示食物名称、数据来源和每 100g 热量。
- 支持减号、加号、直接输入及 100g/150g/200g/一份快捷值。
- 重量允许范围 1–10,000g；小数最多 1 位。
- 实时计算热量、蛋白质、脂肪和碳水。
- 用户必须选择餐次，可填写备注。
- 保存成功后返回对应餐次详情或首页，并刷新汇总。
- 连续点击保存必须防重复提交。

### 6.4 餐次详情（P0）

- 展示餐次总热量、三大营养素和食物列表。
- 支持继续添加、编辑数量、修改餐次、删除记录。
- 删除需要二次确认；成功后立即刷新餐次与首页汇总。
- 规则化建议只能基于已记录数据，数据不足时不显示。

### 6.5 饮食日历（P1）

- 月视图显示有记录的日期标识和选中状态。
- 点击日期加载当日摄入、运动消耗、剩余热量及餐次摘要。
- 支持前后切换月份，默认限制查询最近 12 个月。
- 日期范围接口应一次返回整月摘要，避免逐日发起请求。

### 6.6 今日运动与推荐（P0）

- 展示当天运动总消耗、总时长及运动记录。
- 支持新增、编辑和删除运动记录。
- 新增字段包括运动类型、开始时间、时长、强度、预估消耗和备注。
- 推荐卡展示名称、强度、时长、预计消耗及开始/查看计划入口。
- 无运动记录时展示“添加第一次运动”和一条低强度推荐。

### 6.7 热量与运动趋势（P1）

- 支持 7 天、30 天、90 天切换。
- 主指标为日均净摄入，同时展示平均饮食摄入和平均运动消耗。
- 展示营养素平均达成率和规则化周总结。
- 数据不足 3 天时显示“继续记录后生成趋势”，不输出结论。

### 6.8 个人中心（P1）

- 展示当前体重、目标体重和阶段进度。
- 管理每日热量与营养目标。
- 管理自定义食品、收藏食品和训练计划。
- 收纳 AI 教练、绑定教练、会员码、通知、隐私和关于入口。
- 涉及个人数据的页面必须说明用途并支持删除账号数据。

## 7. 数据模型需求

### 7.1 现有模型调整

**FoodItem 建议新增：**

- `baseAmount decimal(10,2)`
- `baseUnit varchar(20)`
- `servingAmount decimal(10,2)` 可空
- `servingUnit varchar(20)` 可空
- `source varchar(20)`：system / user
- `isFavorite boolean`

**MealRecord 建议调整：**

- 将 `quantity` 明确为用户输入数量，新增 `quantityUnit`。
- 新增 `caloriesSnapshot`、`proteinSnapshot`、`fatSnapshot`、`carbsSnapshot`。
- 新增 `photoUrl` 可空；照片不参与计算。
- 保留 `foodItemId` 关联，但历史汇总使用快照字段。

### 7.2 新增模型

**ExerciseRecord：**

- `id`、`userId`、`exerciseDate`、`exerciseType`
- `startTime`、`durationMinutes`、`intensity`
- `caloriesBurned`、`source`、`note`
- `createdAt`、`updatedAt`

**UserGoal：**

- `userId`、`dailyCalorieGoal`
- `carbsGoal`、`proteinGoal`、`fatGoal`
- `currentWeight`、`targetWeight`、`goalType`
- `updatedAt`

**DailySummary（可实时聚合，后期可缓存）：**

- `date`、`intakeCalories`、`exerciseCalories`
- `remainingCalories`、`netCalories`
- `carbs`、`protein`、`fat`、`recordCount`

## 8. 接口需求

### 8.1 复用与调整现有接口

- `GET /foods`：补充分页、常用/最近/收藏筛选。
- `GET /foods/search?keyword=`：增加分页与空关键词保护。
- `POST /foods`：支持营养基准、每份重量和数据来源。
- `POST /records`：由后端计算并保存营养快照。
- `GET /records?date=`：继续用于餐次详情。
- `GET /records/range`：支持日历明细查询。
- `GET /records/stats/daily`：返回目标、运动、剩余和净摄入。
- `GET /records/stats/weekly`：升级为统一趋势接口。

### 8.2 建议新增接口

| 方法 | 路径 | 用途 | 优先级 |
| --- | --- | --- | --- |
| GET | `/dashboard/today?date=` | 首页一次性聚合数据 | P0 |
| PUT | `/records/{id}` | 编辑饮食记录 | P0 |
| POST | `/photos` | 上传饮食照片，仅留存 | P1 |
| GET/POST | `/exercises` | 查询或新增运动记录 | P0 |
| PUT/DELETE | `/exercises/{id}` | 编辑或删除运动记录 | P0 |
| GET | `/stats/trend?range=7d` | 饮食、运动与净摄入趋势 | P1 |
| GET/PUT | `/users/me/goals` | 查询或修改目标 | P1 |
| GET | `/calendar/summary?month=` | 月历每日摘要 | P1 |

### 8.3 通用接口规则

- 所有用户数据必须按登录用户隔离，禁止信任客户端传入的 `userId`。
- 金额无关数值使用 `BigDecimal`，避免浮点误差。
- 写接口支持幂等键或前端防重标识。
- 错误响应统一包含 `code`、`message`、`requestId` 和字段级错误。
- 日期统一使用 `yyyy-MM-dd`，时间使用带时区的 ISO 8601。

## 9. 非功能需求

### 9.1 性能

- 首页首屏接口 P95 ≤ 800ms，页面可交互时间 ≤ 2 秒。
- 搜索接口 P95 ≤ 500ms。
- 月历与 90 天趋势接口 P95 ≤ 1.2 秒。
- 图片上传前客户端压缩，单张建议 ≤ 2MB。

### 9.2 可靠性与安全

- 用户数据读写必须经过登录鉴权。
- 日志不得记录昵称、体重、备注全文或图片地址中的临时凭证。
- 删除账号后，按隐私策略删除或匿名化饮食、运动和身体数据。
- 对搜索、上传和写接口增加频率限制。

### 9.3 可用性

- 主要点击区域不小于 44×44px。
- 文字颜色满足基础对比度要求，不仅依靠颜色表达状态。
- 网络失败、空数据、加载中和保存成功必须有明确状态。
- 支持常见微信小程序屏幕尺寸及安全区域。

## 10. 埋点与监控

建议埋点事件：

- `home_view`、`record_entry_click`
- `food_search`、`food_search_empty`
- `calorie_calculate`、`meal_record_submit`、`meal_record_success`
- `exercise_add`、`exercise_complete`
- `trend_view`、`goal_update`

每个事件至少包含时间、页面、用户匿名标识、来源入口和结果状态。服务端监控首页聚合接口、搜索接口、写入接口的成功率和延迟。

## 11. 验收标准

### 11.1 P0 核心链路

1. 用户可从首页进入食品搜索，选择鸡胸肉，输入 150g。
2. 当鸡胸肉为 165 千卡/100g 时，页面显示 248 千卡。
3. 保存午餐后，午餐详情和首页摄入同步增加 248 千卡。
4. 用户添加 20 分钟快走后，首页运动消耗更新，但剩余可摄入不增加。
5. 趋势页净摄入按“饮食摄入－运动消耗”计算。
6. 重复点击保存不会产生两条记录。
7. 编辑或删除记录后，所有聚合数据在刷新后保持一致。

### 11.2 质量门槛

- 计算规则单元测试覆盖率 ≥ 90%。
- P0 主流程接口与页面冒烟测试全部通过。
- 无阻断级或严重级缺陷上线。
- 真机覆盖 iOS、Android 各至少两种常见屏幕尺寸。

## 13. 风险与待确认事项

### 13.1 主要风险

- 现有 `FoodItem.calories` 的单位含义不统一，可能导致历史热量计算错误。
- 现有 `MealRecord` 未保存营养快照，食品修改会影响历史统计。
- 首页一次请求多个旧接口可能造成首屏慢和数据不同步。
- 运动消耗属于估算值，需要明确提示，避免被理解为精确医学数据。

### 13.2 开发前必须确认

1. 现有食品库热量是“每份”还是“每 100g”，并完成数据盘点。
2. 历史记录是否迁移为快照，无法识别单位的数据如何处理。
3. V1 是否允许用户手动修改运动消耗。
4. 用户目标来自手动设置还是需要公式推荐。
5. 是否保留现有“历史”Tab；本 PRD 建议由日历和趋势替代。
6. 拍照留存的保存周期、数量限制和删除规则。

## 14. 上线建议

- 先灰度给测试账号和内部用户，重点核对历史数据迁移与热量结果。
- 上线前对同一批测试数据在旧版与新版进行汇总对账。
- 监控首页接口、食品搜索、饮食保存和运动保存四条关键链路。
- 保留数据库迁移回滚脚本和旧版接口至少一个发布周期。

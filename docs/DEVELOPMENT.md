# 饮食与运动小程序开发执行指南

## 1. 目的

本指南把优化版 PRD、UI 设计、现有小程序和 Spring Boot 后端整理成一套可执行的开发流程。

具体阶段任务、验收清单和 Git 交付要求见 [`milestones/README.md`](./milestones/README.md)。

当前代码是可运行的饮食记录原型，不等于优化版最终产品。开发时应逐步替换旧页面和旧接口，不在旧结构上无限叠加功能。

## 2. 开发基准

### 2.0 当前本地启动

当前工程基线统一为 Java 17、MySQL 8.0.46 和 Flyway V1–V8。MySQL 与后端均由本地 Docker 启动，无需宿主机数据库或云服务器：

```bash
bash backend/scripts/check-e1-environment.sh
bash deploy/local/start.sh
```

微信开发者工具只导入 `miniapp/`。develop/trial 标识默认请求当前电脑的局域网地址 `http://192.168.3.25:8080/api`；换网络时用本地 `apiBaseUrl` 覆盖。release 保留未来 HTTPS 地址且不读取本地覆盖。真实微信 Secret 只进入 `deploy/local/.env.local`。完整说明见 [`LOCAL-DEVELOPMENT.md`](./LOCAL-DEVELOPMENT.md)。

V1–V8 是面向空 MySQL 8 库的基线迁移，V6 会补齐基础系统食品，V7 增加运动记录，V8 增加用户目标与匿名删除审计。已有业务数据的旧库不能直接清空或自行开启 `baseline-on-migrate`，必须按 [`mysql8-data-migration.md`](./mysql8-data-migration.md) 执行自动备份、旧结构升级、V5 接管、后续增量升级和 Hibernate 校验。

### 2.1 基准优先级

出现冲突时按以下顺序处理：

1. 已评审的 PRD 业务规则
2. `design/优化版/` 页面与设计说明
3. 本开发指南
4. API 文档
5. 当前旧版代码行为

不能因为旧代码已经存在，就跳过 PRD 中的营养快照、用户隔离、防重复提交和异常状态。

### 2.2 优化版信息架构

一级导航固定为五项：

| 导航 | 对应设计稿 | 主要职责 |
| --- | --- | --- |
| 今日 | `01-首页-热量总览.png` | 今日摄入、目标、运动、餐次摘要 |
| 趋势 | `04-热量与运动趋势.png` | 7/30/90 天摄入、消耗和净摄入 |
| 记录 | `05-食品搜索与记录.png` | 食品搜索、常用、最近、自定义、拍照 |
| 运动 | `03-今日运动与推荐.png` | 运动记录和规则化推荐 |
| 我的 | `08-个人中心.png` | 目标、食品库、计划和设置 |

二级页面：

- `02-热量计算.png`
- `06-饮食日历.png`
- `07-餐次详情.png`

旧版“历史”Tab 不再单独保留，其能力进入“趋势”和“饮食日历”。旧版“统计”Tab 替换为“趋势”。

## 3. 开发前必须完成的决策

以下事项未确认前，不进入大规模页面开发：

- [x] 数据库统一使用 MySQL 8，并删除其他数据库驱动和专用 SQL。
- [x] Java 编译、CI、容器和本机基线统一为 Java 17；本地开发必须显式使用 JDK 17。
- [x] 重量类食品统一按每 100g 保存营养数据，按份食品使用 `baseAmount=1` 并保留原单位。
- [x] 现有按“份”保存的食品数据迁移为 `baseAmount=1` 并保留原单位和历史来源标记。
- [x] 饮食记录保存营养快照，历史统计不重新读取当前食品营养值。
- [x] V1 运动消耗允许用户在规则估算基础上按设备数据手动修正，并保存最终确认值。
- [x] V1 用户热量目标采用手动设置；未设置时首页与趋势使用明确的默认目标。
- [ ] 拍照文件的数量、大小、保存时间和删除策略。
- [x] 生产配置仅从环境变量读取密钥；真实值不进入仓库。历史泄露值仍须在对应平台轮换。

决策完成后，应写入 PRD 或独立 ADR，不只保存在聊天记录中。

M0–M8 已发生的问题、固定防复发规则和阶段收口顺序见 [`DEVELOPMENT-RETROSPECTIVE.md`](./DEVELOPMENT-RETROSPECTIVE.md)。开始新阶段前必须先检查该文档。

## 4. 推荐开发阶段

### Phase 0：工程与数据基础

目标：建立可以安全持续开发的基线。

后端：

- 统一 Java 和 MySQL 8 技术栈。
- 拆分 `local/test/prod` 配置，仓库不保存真实密钥。
- 引入 Flyway，建立第一版数据库迁移。
- 引入 DTO、Jakarta Validation、统一异常和统一错误响应。
- 删除公开 Seed 接口、敏感日志和不安全的请求体日志。
- 修复文件上传类型、大小、路径与访问策略。
- 为用户、食品、饮食记录建立必要索引和约束。

小程序：

- 建立唯一登录 Promise，页面请求统一等待登录完成。
- 401 重新登录使用互斥机制，避免并发登录。
- 建立开发/测试/生产 API 地址切换方式。
- 抽取颜色、字体、间距、圆角、阴影等设计 Token。
- 建立统一的加载、空状态、错误状态和重试组件。
- 删除 Token、openid 和用户资料等敏感日志。

测试与工程：

- 建立后端测试目录和第一批计算规则测试。
- 建立 CI，至少执行编译、单元测试和静态检查。
- 增加 `.editorconfig` 和统一格式化规则。
- 将当前目录整理提交为干净基线。

完成标准：

- 新环境只依赖文档和环境变量即可启动。
- 仓库扫描不到真实密钥。
- 后端构建和核心测试可在 CI 运行。
- 小程序首次启动不会并发登录或把网络错误显示为空数据。

### Phase 1：P0 饮食主链路

目标：完成“今日 → 搜索食品 → 计算热量 → 保存 → 餐次详情”的闭环。

#### 后端顺序

1. 调整 `FoodItem` 营养基准和份量字段。
2. 调整 `MealRecord` 数量单位和营养快照字段。
3. 实现食品搜索、最近、常用和自定义食品接口。
4. 实现饮食记录创建、编辑和删除。
5. 实现 `GET /dashboard/today?date=` 首页聚合接口。
6. 所有写操作增加参数校验、用户隔离和幂等保护。

#### 小程序顺序

1. 实现 `05-食品搜索与记录`。
2. 实现 `02-热量计算`。
3. 实现 `07-餐次详情`。
4. 重构 `01-首页-热量总览`。
5. 接入统一加载、空状态、错误状态和防重复提交。

#### 核心验收

- 165 千卡/100g 的鸡胸肉输入 150g，结果显示 248 千卡。
- 保存记录后，餐次详情和首页同时增加 248 千卡。
- 连续点击保存只产生一条记录。
- 修改食品库数据不会改变历史记录统计。
- 用户不能读取或引用其他用户的自定义食品和饮食记录。

### Phase 2：P0 运动模块

目标：打通运动记录与今日热量状态。

后端：

- 新增 `ExerciseRecord` 和迁移脚本。
- 实现运动查询、新增、编辑和删除接口。
- 实现规则化消耗计算和推荐规则。
- 首页聚合接口增加运动摘要。

小程序：

- 实现 `03-今日运动与推荐`。
- 实现新增、编辑和删除运动流程。
- 首页展示运动时长、消耗和最多一条建议。

核心验收：

- 添加 20 分钟快走后，首页运动消耗立即更新。
- 运动消耗不自动增加剩余可摄入额度。
- 推荐最多展示两项，不输出医学诊断或结果承诺。

### Phase 3：P1 趋势、日历和个人中心

目标：完成优化版剩余页面。

- 实现 `04-热量与运动趋势`，支持 7/30/90 天。
- 实现 `06-饮食日历`，整月摘要一次加载。
- 实现 `08-个人中心` 和用户目标。
- 将食品库、训练计划、服务和设置收纳到个人中心。
- 增加账户数据删除和隐私说明入口。

### Phase 4：本地整体验收

- API 性能、数据库索引和慢查询检查。
- iOS、Android 真机和安全区域适配。
- 上传、搜索和写接口限流。
- 日志脱敏、监控、告警和请求 ID。
- 数据库迁移与回滚演练。
- 本地真实账号验证和新旧统计结果对账。

云端部署、域名、HTTPS、合法域名和体验版灰度在项目全部完成后另开发布阶段，不阻塞 Phase 0–4。

## 5. 目标代码结构

### 5.1 小程序

当前功能较少时保持简单，页面增加后再按分包组织：

```text
miniapp/
├── pages/                 # 五个 TabBar 一级页面
├── package-food/          # 热量计算、餐次详情、饮食日历
├── package-exercise/      # 运动新增和编辑
├── package-user/          # 目标、食品库和设置
├── components/            # 已被多个页面复用的组件
├── services/              # 按业务拆分的 API
├── shared/                # Token、常量、日期和纯工具
├── assets/
├── app.js
└── app.json
```

目录只有在出现真实代码时创建，不预先制造空目录。

### 5.2 后端

运动、目标和趋势加入后，建议按业务功能分包：

```text
com.diettracker/
├── common/
│   ├── config/
│   ├── exception/
│   └── response/
├── auth/
├── user/
├── food/
├── meal/
├── exercise/
├── stats/
└── storage/
```

每个业务包可包含 `controller/service/repository/entity/dto`。不要让一个 Service 同时负责食品、饮食记录、统计、运动和目标。

## 6. 前端开发规范

### 6.1 页面职责

- 页面负责生命周期、交互和展示状态。
- 页面不直接调用 `wx.request`。
- 营养计算规则优先由后端负责，前端计算只用于实时预览。
- 相同计算必须有共享纯函数和测试，不能散落在多个页面。

### 6.2 页面状态

每个异步页面至少区分：

```text
idle / loading / success / empty / error / submitting
```

网络失败不能转换成空数组伪装成“暂无数据”。写操作必须使用 `submitting` 防止重复点击。

### 6.3 UI 规范

- 视觉基准只使用 `design/优化版/`。
- 主色、背景、文字、圆角、阴影和间距集中为 Token。
- 卡片圆角统一，避免页面各自定义近似值。
- 主要点击区域不小于 44×44px。
- 页面必须实现加载、空、错误、禁用和超长文本状态。
- 中英文文案不混用。

### 6.4 登录和请求

- Token 不写入日志。
- 页面数据请求必须等待登录 Promise。
- 同一时刻只允许一个重新登录请求。
- 请求层返回结构化错误，页面根据错误码决定提示或重试。
- 本地真机调试可以关闭域名校验；未来体验/生产环境必须使用 HTTPS 合法域名，不依赖该开关。

## 7. 后端开发规范

### 7.1 接口边界

- Controller 只接收 DTO，不直接接收 JPA Entity。
- Controller 不直接访问 Repository。
- 所有写入 DTO 使用参数校验。
- 用户 ID 只从鉴权上下文获取，不接受客户端传入。
- 错误统一返回 `code`、`message`、`requestId` 和字段错误。

### 7.2 数据规则

- 营养、重量和热量使用 `BigDecimal`。
- 饮食历史使用营养快照。
- 用户数据字段默认不可为空，并建立用户与日期组合索引。
- 数据结构变化只能通过 Flyway 脚本完成。
- 示例数据只允许在本地或测试 Profile 初始化。

### 7.3 安全

- 配置文件不提供真实密钥默认值。
- 日志不记录 Token、微信登录响应、昵称、备注全文和上传临时凭证。
- 上传文件校验真实内容并重新生成安全文件名。
- 搜索、上传、登录和写接口设置频率限制。
- Seed、调试和诊断接口不能暴露在生产环境。

## 8. API 协作流程

每个功能按以下顺序开发：

1. PRD 确认输入、输出和异常。
2. 后端定义 Request/Response DTO 和 OpenAPI。
3. 前端根据接口契约建立 Service。
4. 后端完成单元测试和接口测试。
5. 前端完成正常、空、加载、错误和重复提交场景。
6. 联调通过后同步更新 `docs/api.md`。

禁止先让前端猜测 JSON，再通过联调反复修改字段。

## 9. Git 工作流

- 仓库唯一远端为 `https://github.com/Lanlnn/diet-tracker.git`。
- 详细操作、分支命名、阶段标签和异常处理见 [`git-workflow.md`](./git-workflow.md)。
- `main/master` 保持可构建。
- M0–M10 阶段分支统一使用 `codex/mN-阶段名`。
- 阶段外功能使用 `codex/feat-模块-功能`，修复使用 `codex/fix-模块-问题`。
- 一次提交只包含一个清晰目的。
- 数据迁移、后端实体、接口 DTO、前端 Service 和文档尽量在同一功能 PR 中完成。
- 大型设计源文件和重复 ZIP 不直接进入普通 Git 历史；需要版本管理时使用 Git LFS。

推荐提交类型：

```text
feat: 新功能
fix: 缺陷修复
refactor: 不改变行为的重构
test: 测试
docs: 文档
chore: 工程配置
```

## 10. 测试策略

### 10.1 后端

- 营养和热量计算：单元测试。
- 用户数据隔离：Repository/Service 集成测试。
- 登录、记录、编辑、删除：Controller 接口测试。
- MySQL 8 兼容性：单元与接口测试使用 H2 MySQL 兼容模式；Flyway 迁移必须另外在真实 MySQL 8 实例上执行，CI 使用 `mysql:8.0` 服务验证空库迁移和 Hibernate Schema Validation。

### 10.2 小程序

- 日期、营养计算和数据转换：纯函数测试。
- 登录、401 和重试：请求层测试。
- 首页、记录、运动：开发者工具与真机冒烟测试。
- 至少覆盖 iOS、Android 各两种常见屏幕尺寸。

### 10.3 核心回归用例

- 首次登录和 Token 过期。
- 150g 食物营养计算。
- 重复点击保存。
- 编辑或删除后首页、餐次和趋势一致。
- 空食品搜索、网络失败和服务器错误。
- 用户之间数据不可见。
- 图片类型、大小和上传失败。

## 11. 完成定义（Definition of Done）

一个功能只有同时满足以下条件才算完成：

- 符合 PRD 和优化版 UI。
- 正常、加载、空、错误和禁用状态齐全。
- 后端完成 DTO、校验、鉴权和用户隔离。
- 关键业务规则有自动化测试。
- 前后端字段和 API 文档一致。
- 无敏感日志、真实密钥或临时调试接口。
- 构建和 CI 通过。
- 在开发者工具和本地同网 iOS、Android 真机完成相关回归。

## 12. 当前任务

M0–M10 已完成代码基线。当前只按 [`NEXT-DEVELOPMENT.md`](./NEXT-DEVELOPMENT.md) 执行 E1：本地环境 → 真实微信登录和基础食品 → 饮食/运动/趋势/日历/目标主链路 → iOS/Android 本地回归。项目完成前不穿插云端部署工作。
